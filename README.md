# PagentZ Atlas

Internal admin console for the PagentZ SaaS. Flutter Web app that shares the `pagentz` Firebase project with the main customer app, but authenticates staff through the dedicated `Atlas-Staff-2o08x` Identity Platform tenant. TOTP MFA is required.

**Repo:** https://github.com/ramavpagentz/PAGENTZ_ATLAS
**Production URL:** https://atlas.pagentz.com (after Cloudflare CNAME is wired)
**Hosting site:** `atlas-13fd3` (`https://atlas-13fd3.web.app`)
**Runtime:** Flutter Web only.

---

## First-time setup

Required once per developer machine:

```bash
# 1. Install FlutterFire CLI if you don't have it
dart pub global activate flutterfire_cli

# 2. Wire Firebase to this project (choose 'Web' when prompted, project = pagentz)
flutterfire configure --project=pagentz

# 3. Install Dart/Flutter packages
flutter pub get
```

Step 2 replaces `lib/core/config/firebase_options.dart` with real credentials for the `pagentz` project. The generated file is **safe to commit** ‚Äî Firebase Web API keys are public by design; access control is enforced by Firestore security rules + the staff tenant + the `isStaff` flag.

---

## Local development

```bash
flutter run -d chrome
```

Sign in with your `@pagentz.com` staff account. First-time users will be walked through TOTP enrollment, then signed out to re-sign-in with their new authenticator code. All staff accounts live in the `Atlas-Staff-2o08x` tenant ‚Äî not in the default (customer) Firebase Auth tenant.

---

## Deploy

```bash
flutter build web --release
firebase deploy --only hosting:atlas --project pagentz
```

The hosting target `atlas` is wired to Firebase Hosting site `atlas-13fd3` via `.firebaserc`. Custom domain `atlas.pagentz.com` is bound to that site via Cloudflare CNAME + Firebase custom-domain verification (Phase C of the build plan, not yet wired).

## Adding a new staff user

1. Identity Platform console ‚Üí scope to `Atlas-Staff` tenant ‚Üí **Users** ‚Üí **Add user** with the `@pagentz.com` email.
2. Copy the UID from the new user row.
3. Firestore Console ‚Üí `users/{uid}` ‚Üí create doc with these fields:

| Field | Type | Value |
|---|---|---|
| `email` | string | the user's email |
| `displayName` | string | full name |
| `isStaff` | boolean | `true` |
| `staffRole` | string | one of `l1_support`, `l2_support`, `engineer`, `admin`, `owner` |
| `staffTenantId` | string | `Atlas-Staff-2o08x` |
| `staffJoinedAt` | timestamp | now |
| `mfaEnrolled` | boolean | `false` |
| `disabled` | boolean | `false` |

4. Send the user the temporary password. They sign in at https://atlas.pagentz.com, get walked through TOTP enrollment, sign in again with their authenticator.

## Security rules

This repo does not own `firestore.rules` ‚Äî the main PAGENTZDEV-dev repo does (rules are global per Firebase project). See [`firestore.rules.atlas-patch`](firestore.rules.atlas-patch) for the helpers and match blocks that must be merged into the main repo's rules and deployed before Atlas can operate safely.

## Architecture notes

- **Identity Platform multi-tenancy.** Customer auth lives in the default tenant (Apple/Email/Google). Staff auth lives in `Atlas-Staff-2o08x` (Email/Password only, TOTP MFA enforced). `FirebaseAuth.instance.tenantId` is pinned at app boot in `StaffAuthService.configureTenant()`. Customer credentials cannot authenticate against Atlas; staff credentials cannot authenticate against the customer app.
- **Customer impersonation (Phase 4)** ‚Äî implemented via a Cloud Function that mints a custom token in the **default (customer) tenant** for the target customer UID, on behalf of the requesting staff user (verified via their staff-tenant ID token). The customer app redeems that token in a new tab with a persistent staff banner. Read-only mode in v1; write mode deferred to v2.
- **Staff audit log** is a Firestore collection (`staff_audit_logs`) with rules `allow update, delete: if false`. Writes happen only through Cloud Functions that perform the action + log atomically.
- **Customer activity logs** live in Azure Postgres (existing) ‚Äî Atlas reads them via an internal API on the main customer-app backend (TBD endpoint). The Atlas Customer Activity Timeline merges Postgres customer activity + Firestore staff audit chronologically. See the build-plan log-architecture spec (separate doc) for the production-grade design.
